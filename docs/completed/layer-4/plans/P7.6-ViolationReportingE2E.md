### Test 1: TestE2E_ViolationReporting_TaintViolationFlow

**Given**: 
- sys:security detecting taint violations (arch-v1.md L286)
- sys:observability with dead-letter queue running (L459)
- DMZ agent attempting to send `INNER_ONLY` tainted data to outer boundary

**When**: DMZ agent sends mail with `INNER_ONLY` taint to outer agent → security detects violation  
**Then**:
- Security emits `taint_violation` event with violation details (L286)
- Event routed to dead-letter queue via Communication layer (L606)
- sys:observability receives `taint_violation` from dead-letter
- Violation stored with: source agent, target, forbidden taint, timestamp, correlationId
- Metrics updated: `taint_violations_total` incremented, `taint_violations_by_type{type="INNER_ONLY"}` incremented
- Violation queryable via `sys:observability.query(traces, filters)` (L490)
- Full chain: violation → dead-letter → observability → metrics verified
**Expected Result**: Taint violation flows through complete reporting chain end-to-end  
**Spec Reference**: arch-v1.md L286 (violation to dead-letter), L459 (sys:observability), L606 (dead-letter queue), L490 (query interface)

### Test 2: TestE2E_ViolationReporting_MultiTypeAggregation

**Given**: 
- sys:observability with metrics collection (arch-v1.md L459)
- Multiple agents generating different violation types:
  - Agent A: taint violation (INNER_ONLY crossing boundary)
  - Agent B: boundary violation (outer→inner without mediation)
  - Agent C: isolation violation (cross-namespace access attempt)

**When**: All three agents trigger violations within same time window  
**Then**:
- Each violation emitted as `taint_violation` event to dead-letter (L286)
- sys:observability receives all three violations
- Violations categorized by type in metrics:
  - `taint_violations_by_type{type="INNER_ONLY"}` = 1
  - `taint_violations_by_type{type="BOUNDARY"}` = 1
  - `taint_violations_by_type{type="ISOLATION"}` = 1
- `taint_violations_total` = 3
- Violations deduplicated via `correlationId` if retries occur (L605)
- Query returns all violations with proper filtering:
  - `query(traces, filters={type: "INNER_ONLY"})` returns Agent A's violation
  - `query(traces, filters={source: "agent:B"})` returns Agent B's violation
- At-least-once delivery verified (no violations lost)
**Expected Result**: Multiple violation types aggregated and queryable in observability  
**Spec Reference**: arch-v1.md L286 (violation emission), L459 (sys:observability), L604-606 (delivery guarantees), L490 (query interface)
