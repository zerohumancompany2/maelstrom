## Parent Requirements

From `arch-v1.md L261-270` - Boundary Model:
1. Inner: Full read/write to sensitive application-context slices (secrets, internal DBs, production APIs). Prompt injection surface is zero because no untrusted Messages ever enter an inner Chart.
2. DMZ: Mediated access. Tools/sub-agents are wrapped; outputs are sanitized (PII redaction, length limits, schema validation). ContextMap blocks marked with forbidden taints are stripped before any LLM call.
3. Outer: Untrusted ingress only (user chat, webhooks, mail from external). Can only talk to DMZ agents via mail; cannot directly read/write inner data.

From `arch-v1.md L290` (Mail-only transitions):
- Transitions across boundaries are explicit and always go through the Mail system (never direct event passing).

## Acceptance Criteria

1. ValidateBoundary accepts inner, dmz, outer as valid boundary types
2. ValidateBoundary rejects unknown or empty boundary types with error
3. Inner boundary has full read/write capabilities and can access secrets
4. DMZ boundary has mediated access capabilities requiring sanitization
5. Outer boundary has ingress-only capabilities, cannot access inner data
