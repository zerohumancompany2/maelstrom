## Parent Requirements

From `arch-v1.md L288-302` - Boundary Enforcement Workflow:
1. Transitions across boundaries are explicit and always go through Mail system (never direct event passing) (L290)
2. Enforced by `sys:security` System Service Chart at runtime (L291)
3. Security service calls `prepareContextForBoundary(runtimeId, boundary)` inside every `assembleContextMap` (L293)
4. Any block containing forbidden taint for current boundary is redacted or dropped before LLM call (L294)
5. Per-block `taintPolicy` overrides global settings (L295)

From `arch-v1.md L261-270` - Boundary Model:
- **inner**: Full read/write to sensitive data; no untrusted Messages ever enter (L268)
- **DMZ**: Mediated access; outputs sanitized; forbidden taints stripped before LLM (L269)
- **outer**: Untrusted ingress only; can only talk to DMZ via mail (L270)

From `arch-v1.md L1320-1322` (SHALL NOT statements):
- SHALL NOT allow inner data to reach outer/DMZ LLM prompts without redaction
- SHALL NOT permit boundary crossing via Events (only Mail)

## Test Descriptions

### Test 1: TestE2E_BoundaryEnforcement_ForbiddentransitionBlocked

**Given**: 
- Inner agent with access to `INNER_ONLY` tainted data (arch-v1.md L268)
- Outer agent attempting to receive mail from inner agent
- sys:security and sys:observability running

**When**: Inner agent attempts to send mail containing `INNER_ONLY` tainted data to outer agent  
**Then**:
- Mail routed through sys:security for validation (L291)
- Security detects forbidden boundary transition (inner→outer with INNER_ONLY taint)
- Mail delivery blocked; outer agent does not receive message
- `taint_violation` event emitted to dead-letter queue (L286)
- sys:observability receives violation with details (source, target, forbidden taint)
**Expected Result**: Forbidden boundary transition blocked with violation reported end-to-end  
**Spec Reference**: arch-v1.md L288-291 (boundary enforcement), L284 (inner→DMZ/outer strip/block), L1321 (SHALL NOT permit boundary crossing violations)

### Test 2: TestE2E_BoundaryEnforcement_ContextMapFiltering

**Given**: 
- DMZ agent with ContextMap containing multiple ContextBlocks (arch-v1.md L205-223)
- One ContextBlock has `INNER_ONLY` taint with `redactMode: redact` policy (L217-220)
- Another ContextBlock has `PII` taint with `redactMode: dropBlock` policy
- Agent in DMZ boundary attempting LLM call

**When**: Agent assembles ContextMap for LLM call → `prepareContextForBoundary` invoked (L293)  
**Then**:
- `prepareContextForBoundary` called with runtimeId and DMZ boundary (L293)
- ContextBlock with `INNER_ONLY` taint is redacted per policy (L294)
- ContextBlock with `PII` taint is dropped entirely (dropBlock mode)
- LLM receives prompt without forbidden tainted content
- Redacted blocks show `[REDACTED]` or equivalent per redactRules (L219-220)
- No `INNER_ONLY` or `PII` data in final assembled prompt
**Expected Result**: ContextMap assembly enforces boundary filtering with per-block taintPolicy  
**Spec Reference**: arch-v1.md L292-296 (ContextMap enforcement), L216-223 (per-block taintPolicy), L1314 (SHALL redact/drop/audit per-block taintPolicy)
